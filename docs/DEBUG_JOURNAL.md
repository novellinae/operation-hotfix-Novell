## Bug 1 — Silent RLS Block

| Field | Entry |
|------|------|
| **Symptom** | The dashboard table displays **“Empty table”** even though there are **5 rows of data in the `shipments` table** in the Supabase database. |
| **Hypothesis** | Row Level Security (RLS) is enabled on the `shipments` table, but there is **no policy allowing the `anon` role to read data** from the table. Because the dashboard uses the public Supabase `anon` key, the query returns an empty result set.<br><br>**Investigation:** After verifying that the database actually contains shipment records in Supabase, the next step was checking the database policies defined in `legacy_setup.sql`. The table had RLS enabled but only defined an **UPDATE policy**, with no policy allowing SELECT access for the `anon` role. Since Supabase enforces RLS strictly, any query executed by the frontend using the anon key would return an empty result set even though data exists. |
| **AI Prompt** | I am debugging a Next.js logistics dashboard that uses Supabase as the backend database. The dashboard loads a table of shipments on the `/dashboard` page, but the table shows “Empty table” even though the `shipments` table in Supabase contains 5 rows. There are no errors in the browser console. Since Supabase Row Level Security is enabled, could the issue be caused by the `anon` role not having permission to read the table? How can I verify this and configure the correct SELECT policy while keeping RLS enabled? |
| **Fix** | Added a **SELECT policy for the `anon` role** on the `shipments` table so the dashboard can read shipment records from Supabase. |

## Bug 2 — Ghost Mutation

| Field | Entry |
|------|------|
| **Symptom** | Changing a shipment status shows a **success toast notification**, but after refreshing the page the shipment status returns to its original value. Inspecting the `shipments` table in Supabase confirms that **no rows were updated**. |
| **Hypothesis** | The Supabase update query might not execute correctly, the server action might return success before the database mutation completes, the update query might not target the correct row, or the database might reject the update due to constraints or triggers.<br><br>**Investigation:** The issue was reproduced by updating a shipment status in the dashboard and refreshing the page, which confirmed that the database value did not change. Inspecting the server action responsible for updating the status showed that the Supabase `.update()` call was executed without `await`. Because the mutation is asynchronous, the server action returned a success response before the database operation completed, causing the UI to display a success toast even though the update had not yet persisted. |
| **AI Prompt** | I am debugging a server action in a Next.js application that updates shipment status in a Supabase database. When the user updates the status, the UI shows a success toast notification, but after refreshing the page the status returns to the original value and the database shows no change. The update logic uses a Supabase `.update()` query inside a server action. Could this issue occur if the database mutation is asynchronous but not awaited? Why would adding `await` allow the update to persist while removing it causes the change to disappear? |
| **Fix** | Added `await` to ensure the **database mutation completes before returning a success response**. |

## Bug 3 — Infinite Loop

| Field | Entry |
|------|------|
| **Symptom** | The dashboard becomes **unresponsive immediately after loading** and users cannot interact with any UI elements. Network inspection shows **hundreds of `GET /dashboard` requests per second** originating from `data-table.tsx`. |
| **Hypothesis** | A client-side navigation loop may be occurring due to a `useEffect` dependency repeatedly triggering `router.push`. Because navigation updates the URL, `searchParams` may be recreated on each render and cause the effect to run repeatedly.<br><br>**Investigation:** The issue was reproduced by loading the `/dashboard` page and observing that the UI immediately froze. Inspecting the browser Network panel showed hundreds of repeated `GET /dashboard` requests being triggered continuously. The request initiator pointed to `data-table.tsx`, specifically inside a `useEffect` hook that updates the URL using `router.push`. Further inspection of the dependency array showed that the effect depends on `table.getState().sorting`, `searchParams`, and `router`. Because `router.push` updates the URL, the `searchParams` object is recreated on every navigation, which causes the `useEffect` to run again. This creates a feedback loop where navigation triggers a rerender, which then retriggers the effect and causes another navigation.|
| **AI Prompt** | I am debugging a Next.js 15 logistics dashboard that uses TanStack Table and Supabase. The issue occurs in the `DataTable` client component where the page freezes immediately after loading and the Network panel shows hundreds of `GET /dashboard` requests per second, even though the user is not interacting with the page. The requests originate from a `useEffect` that updates the URL using `router.push` based on sorting state. The dependency array currently includes `table.getState().sorting`, `searchParams`, and `router`. I suspect one of these dependencies is causing the effect to rerun repeatedly and create a navigation loop. How can I determine which dependency is triggering the rerenders and what would be the correct dependency configuration to prevent the infinite loop while still updating the URL when sorting changes? |
| **Fix** | Updated the dependency array to depend only on the stable React state `sorting`. This prevents repeated navigation and stops the infinite request loop. |

Because the TanStack Table cell renderer attempts to access `cargo.item` and `cargo.weight_kg`, the values appear undefined when the value is actually an array, resulting in an empty Cargo column.<br><br> The repository already includes a utility function `normalizeCargoDetails` designed to convert the JSON array format into a single object. However, the data returned from Supabase is passed directly to the table without validating or normalizing the data shape first. As a result, the UI receives data that does not match the expected TypeScript type.|
| **AI Prompt** | I am debugging a Next.js logistics dashboard using Supabase. The `shipments` table contains a `cargo_details` column stored as JSONB. The dashboard loads correctly and displays all shipment rows, but the Cargo column appears empty even though the database contains cargo information. The TypeScript type defines `cargo_details` as an object with `item` and `weight_kg`, but inspecting the database shows the values stored as JSON arrays. ## Bug 5 — The Unreliable Search

| Field | Entry |
|------|------|
| **Symptom** | When QA uses the search box to filter shipments by cargo item name, the results sometimes do not match the entered query. For example, typing "Medical" may return all shipments, or typing "Electronics" may display "Medical Supplies". The issue occurs more frequently when the user types quickly or when the network is slow (e.g., throttled to 3G). In some cases, clearing the search input does not immediately restore the full dataset, and the table may temporarily display results from a previous query. |
| **Hypothesis** | The search input triggers a server action for every keystroke, creating multiple asynchronous requests running at the same time. Under slow network conditions, these requests may complete out of order. When an older request finishes after a newer one, its response can overwrite the correct results in the table state. This results in inconsistent search results where the displayed rows do not match the current search input. <br><br> **Investigation:** The issue was reproduced by typing quickly in the search input while enabling network throttling (3G) in Chrome DevTools. The Network panel showed multiple requests being sent to the `/dashboard` route through the `searchShipments` server action. Because the search function was triggered on every keystroke, several requests were active simultaneously. Under slow network conditions, earlier requests sometimes finished after later ones, which caused outdated results to overwrite the correct table data. Additional testing also revealed that when the search input was cleared, previous responses could still update the table after the reset occurred. This confirmed that the root cause was a race condition between asynchronous search requests.|
| **AI Prompt** | I am debugging a Next.js logistics dashboard using Supabase. When QA uses the search input to filter shipments by cargo item name, the results sometimes do not match the entered query. For example, typing "Medical" may return all shipments or show results for a different item. The issue becomes much more noticeable when the network connection is slow (for example when throttled to 3G in Chrome DevTools). The search functionality calls a server action on every keystroke. Could this behavior be caused by asynchronous requests completing out of order? Why would slower networks make the issue more visible, and what is the best way to ensure that only the most recent search query updates the table results? |
| **Fix** | Implemented a debounced search mechanism in the client component using a `useDebounce` hook so that the server action is only triggered after the user stops typing briefly. Additionally, a request guard using `useRef` was introduced to track the latest query and ignore outdated responses. This ensures that only the response corresponding to the most recent query updates the table state. When the search input is empty, the table is reset to the original dataset. |

## Bug 6 — The Persistent Ghost

| Field | Entry |
|------|------|
| **Symptom** |QA reported that when changing a shipment status from “Delivered” back to “Pending”, the UI displays a green success notification indicating the update succeeded. However, after refreshing the page (F5), the shipment status remains “Delivered” and the value in the Supabase database never changes. QA also confirmed that all other status transitions work correctly, such as Pending → In Transit and In Transit → Delivered.|
| **Hypothesis** |Since all other transitions persist correctly after fixing the previous mutation issue, the server action responsible for updating shipment status is likely functioning as expected. Because the issue occurs only for one specific transition (Delivered → Pending), the cause is likely related to a database-level constraint or trigger that blocks this particular status change.<br><br> **Investigation:** I reviewed the database setup in legacy_setup.sql and found a trigger function prevent_delivered_to_pending() that checks whether the previous status (OLD.status) is Delivered and the new status (NEW.status) is Pending. If this condition is met, the trigger raises a PostgreSQL exception: "Cannot revert a delivered shipment to pending status". This suggests that when the application attempts to update the status from Delivered to Pending, the database rejects the update entirely, leaving the original status unchanged even though the UI displays a success notification.|
| **AI Prompt** |I am debugging a Next.js logistics dashboard using Supabase. When QA attempts to change a shipment status from Delivered back to Pending, the UI shows a green success notification but after refreshing the page the status remains Delivered and the database record never changes. Other transitions such as Pending → In Transit work correctly. While investigating the database schema I found a trigger function prevent_delivered_to_pending() that raises a PostgreSQL exception if OLD.status = 'Delivered' and NEW.status = 'Pending'. Could this trigger be causing the update query to fail silently, and if so how should the server action detect and handle the Supabase error so the UI shows a proper error message instead of a false success notification?|
| **Fix** |The PostgreSQL trigger enforcing the business rule was intentionally kept because it prevents invalid state transitions in the logistics workflow. Instead of removing the trigger, the application layer was updated to properly handle database errors returned by Supabase. The updateShipmentStatus server action now checks the error returned by the update query and returns a structured failure response when the trigger blocks the update. The client component then displays an error toast instead of a success notification when the Delivered → Pending transition is attempted. This ensures the UI accurately reflects the database state and prevents misleading success messages.|
